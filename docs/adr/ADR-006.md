# ADR-006: 수집 상한을 코드 주석이 아닌 config로 둔 이유

## Context
개발/테스트/운영 환경마다 API 호출량, 기간 범위, 페이지 크기 같은 파라미터 요구가 다릅니다.  
값을 코드 내부 상수로 관리하면 환경 전환 시 코드 수정/리뷰/배포가 반복되어 운영 민첩성이 떨어집니다.

## Decision
수집 상한과 정책 값을 `configs/pipeline.yaml`, `dev.yaml`, `prod.yaml`로 외부화하고,  
`--config` 인자로 런타임 주입하는 구조를 기본으로 채택했습니다.

## Consequences
- 장점:
  - 환경별 정책 전환이 코드 변경 없이 가능합니다.
  - 실험/운영 파라미터를 명시적으로 문서화할 수 있습니다.
  - 리뷰 포인트가 "로직"과 "운영 값"으로 분리되어 협업 효율이 올라갑니다.
- 단점:
  - 설정 파일 간 불일치/드리프트 관리가 필요합니다.
  - 잘못된 config 배포 시 런타임 동작이 즉시 변할 수 있습니다.

## Alternatives Considered
- 코드 상수/주석 기반 운영:
  - 단일 환경에서는 단순하지만, 멀티 환경 운영에서 변경 비용이 높아 비효율적이라고 판단했습니다.
- 환경변수만으로 전부 제어:
  - 비밀값에는 적합하나 복잡한 구조형 파라미터 관리와 가독성이 떨어져 보조 수단으로 제한했습니다.
