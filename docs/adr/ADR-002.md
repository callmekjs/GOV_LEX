# ADR-002: content_hash를 본문이 아닌 ID 해시로 정의한 이유

## Context
연동한 공개 API들은 소스별로 본문 제공 방식이 다르고, 일부는 본문이 요약/부분 텍스트만 제공되거나 구조가 불안정합니다.  
초기 단계에서 본문 해시를 강제하면 수집 성공률과 중복 판정 일관성이 급격히 떨어질 위험이 있었습니다.

## Decision
MVP 단계의 `content_hash`는 `(제목, 발행일, 소스 식별자)` 정규화 결합값의 SHA-256으로 정의했습니다.  
즉 "본문 해시"가 아닌 "식별 메타데이터 해시"를 표준으로 채택했습니다.

## Consequences
- 장점:
  - 소스별 본문 품질 편차와 무관하게 안정적인 dedup 기준을 확보합니다.
  - KR/US/국회 소스 간 공통 규칙을 유지할 수 있습니다.
  - 파싱 실패/본문 누락 상황에서도 파이프라인 일관성이 유지됩니다.
- 단점:
  - 실질적으로 같은 본문이더라도 식별 메타가 달라지면 중복으로 잡히지 않을 수 있습니다.
  - 본문 수정 이력 추적에는 별도 콘텐츠 지문 체계가 필요합니다.

## Alternatives Considered
- 본문 텍스트 SHA-256을 즉시 도입:
  - 장기적으로는 더 정확한 dedup 가능성이 있으나, 초기 소스 편차가 커 실패율과 예외 처리가 증가해 MVP 안정성을 해친다고 판단했습니다.
- 소스별 서로 다른 해시 전략:
  - 단기 우회는 가능하나 스키마 일관성이 깨져 유지보수 비용이 증가하므로 채택하지 않았습니다.
