# ADR-001: 왜 RDB 대신 JSONL로 시작했는가

## Context
GovLex-Ops 초기 목표는 "법률 데이터 ETL 파이프라인이 끝까지 동작함"을 빠르게 증명하는 것이었습니다.  
초기 단계에서 스키마 변경, 수집 실패 재실행, QA 룰 튜닝이 반복될 가능성이 높았고, 운영 복잡도보다 개발 속도와 추적 가능성이 더 중요했습니다.

## Decision
초기 저장소 기본값을 RDB가 아닌 JSONL(`data_index/normalized/docs.jsonl`)로 채택했습니다.  
동시에 향후 확장을 위해 저장소를 어댑터 패턴(`DocumentStore`)으로 분리해 SQLite/pgvector 전환 경로를 열어두었습니다.

## Consequences
- 장점:
  - 파일 단위 diff/백업/복구가 쉬워 디버깅과 회귀 분석이 빠릅니다.
  - 로컬 재현이 단순해 신규 환경 온보딩 비용이 낮습니다.
  - run 단위 산출물 확인이 직관적이라 면접/데모에 유리합니다.
- 단점:
  - 복잡한 필터/집계 쿼리 성능은 RDB 대비 제한적입니다.
  - 동시성 제어와 인덱싱 기능은 별도 설계가 필요합니다.

## Alternatives Considered
- PostgreSQL/pgvector를 초기부터 기본 저장소로 사용:
  - 장점은 강력한 쿼리/인덱스이지만, MVP 단계에서 인프라 의존성이 커져 개발 속도와 재현성이 저하된다고 판단해 보류했습니다.
- SQLite를 초기 기본값으로 사용:
  - JSONL보다 쿼리는 강하지만, 초기 스키마 실험 속도와 파일 가시성 측면에서 JSONL이 더 적합하다고 판단했습니다.
