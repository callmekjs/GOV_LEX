# ADR-005: run_id 격리와 normalized 누적을 분리한 이유

## Context
파이프라인 운영에서는 "이번 실행의 상태"와 "장기 누적 자산"을 동시에 관리해야 합니다.  
둘을 같은 위치/수명주기로 다루면 재실행 비교, 장애 분석, 회귀 점검이 어려워집니다.

## Decision
`runs/<run_id>/`에는 실행 단위 산출물(events, metrics, quality reports)을 격리 보관하고,  
`data_index/normalized/`에는 누적 통합 문서/중복기억을 저장하는 이중 구조를 채택했습니다.

## Consequences
- 장점:
  - 실행 간 비교 및 장애 원인 추적이 쉬워집니다.
  - 누적 저장소를 안정적으로 유지하면서도 실험 실행을 분리할 수 있습니다.
  - Replay 기능에서 특정 run 재검증이 가능해집니다.
- 단점:
  - 디렉터리 구조가 단순 단일 저장 방식보다 복잡합니다.
  - 저장량이 증가해 장기 보관 정책(압축/보존 기간)이 필요합니다.

## Alternatives Considered
- 실행 결과와 누적 결과를 단일 디렉터리로 통합:
  - 구조는 단순하지만 run 재현성과 변경 영향 분석이 약화되어 운영 품질이 떨어진다고 판단했습니다.
- run_id 없이 최신 결과만 유지:
  - 저장 공간은 절약되나 회귀 분석/감사 가능성이 사실상 사라져 채택하지 않았습니다.
