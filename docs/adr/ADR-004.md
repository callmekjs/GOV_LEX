# ADR-004: QA를 ETL 외부 단계로 분리한 이유

## Context
법률 데이터 파이프라인은 "수집 성공"만으로 충분하지 않고, 중복/결측/날짜 이상 같은 품질 이슈를 별도 추적해야 합니다.  
초기 MVP에서 QA를 ETL 내부 분기 로직에 섞어두면 실패 원인 추적과 룰 확장이 어려워졌습니다.

## Decision
QA를 `QARuleEngine` + 리포트 생성(`quality_failures.jsonl`, `quality_report.md`)으로 분리된 단계로 설계했습니다.  
또한 Great Expectations 같은 대형 프레임워크는 초기 도입하지 않고, 도메인 맞춤 룰(R01/R02/R05/R07)을 코드로 직접 관리합니다.

## Consequences
- 장점:
  - 실패 원인과 격리 기준이 명확해 운영 가시성이 높아집니다.
  - 룰 추가/수정 시 ETL 수집 코드와 결합도가 낮아 변경 리스크가 줄어듭니다.
  - run 단위 보고서로 품질 상태를 즉시 설명할 수 있습니다.
- 단점:
  - 룰 수가 늘어나면 자체 QA 프레임워크 유지비가 증가합니다.
  - 표준화된 외부 생태계 기능(풍부한 프로파일링 등)은 즉시 사용하기 어렵습니다.

## Alternatives Considered
- ETL 단계 내부에서 if/else로 품질 검사 수행:
  - 구현은 빠르나 디버깅/관측/룰 확장성이 낮아 장기 운영에 부적합하다고 판단했습니다.
- Great Expectations 즉시 도입:
  - 기능은 강력하지만 MVP 범위 대비 학습/운영 오버헤드가 커 우선순위를 낮췄습니다.
