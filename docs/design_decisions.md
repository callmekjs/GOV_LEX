# 설계 결정 기록 (Design Decisions)

> "왜 이렇게 만들었는가"를 기록합니다.
> 면접·코드 리뷰에서 설계 의도를 설명하기 위한 문서입니다.

---

## D1. 왜 모든 실행에 run_id를 붙이는가?

**문제**: "어제 실행이랑 오늘 실행이 왜 다른 결과가 나왔지?"를 추적할 수 없으면 품질 관리 불가능.

**결정**: 매 실행마다 타임스탬프 기반 고유 폴더 생성. 모든 산출물을 그 안에 저장.

**결과**:
- `runs/run_20260419_122630_b2326c/events.jsonl` — 그 실행에서 벌어진 일
- `runs/run_20260419_122630_b2326c/quality_report.md` — 그 실행의 품질
- 언제든 특정 실행 재현 가능

**대안 고려**: 단일 `latest/` 폴더 → 기각 (과거 실행 사라짐)

## D2. 왜 표준 필드를 9개만으로 제한했는가?

**문제**: 필드가 많을수록 KR·US 모두 채우기 어려워짐 → 결측률 상승.

**결정**: 필수 9개만 공통화. 국가별 추가 정보는 `metadata` 자루에.

**원칙**: `metadata`의 반복 키는 다음 버전에서 정식 필드로 승격.

**대안 고려**: 모든 필드를 표준화 → 기각 (3개국 이상으로 확장 시 유지 불가)

## D3. 왜 Congress.gov API를 검색이 아닌 직접 지정 방식으로 바꿨는가?

**문제**: `query="artificial intelligence"` 검색이 "Providing for adjournment..." 같은 관련 없는 법안을 반환.

**발견**: Congress.gov API의 `query` 파라미터는 제목뿐 아니라 모든 필드에서 검색함.

**결정**: 실제 AI·데이터 관련 법안 번호를 직접 지정하는 방식으로 변경.

**효과**: 
- 이전: "Providing for adjournment" 같은 절차 법안 반환
- 이후: "Algorithmic Accountability Act", "AI Accountability Act" 반환

## D4. 왜 중복 감지를 제목이 아닌 content_hash로 하는가?

**문제**: 제목이 같아도 내용이 다를 수 있고, 공백·유니코드 차이로 같은 문서가 다른 문서로 인식.

**결정**: SHA-256 해시를 사용하되, 해시 생성 전 텍스트 정규화 적용.
- 앞뒤 공백 제거
- 유니코드 NFC 정규화
- 연속 공백 단일화

**효과**: 공백 하나 차이로 중복 감지 실패하던 문제 해결.

## D5. 왜 LLM 답변 생성을 뺐는가?

**포지셔닝 이유**: Legal Data Specialist 포지션은 AI 엔진이 아닌 **데이터 인프라**를 담당. LLM 답변 품질은 데이터 품질과 별개의 변수.

**기술적 이유**: LLM 답변이 틀렸을 때 "데이터 문제인가, 모델 문제인가" 구분이 어려워짐.

**대신**: 검색 결과에 **원문 링크 강제 첨부**로 "근거 있는 답변" 원칙 유지.

**Week 3+ 계획**: 파이프라인 안정화 후 LLM 레이어 추가. 이때 "내 데이터가 Chat CODIT에 투입 가능한 형태"임을 증명.

## D6. 왜 외부 DB 없이 로컬 파일(JSONL)로 저장하는가?

**결정**: MVP는 Postgres·Neo4j 없이 JSONL + FAISS 파일만으로 동작.

**이유**:
- 인프라 0으로 재현성 보장 (git clone 후 즉시 실행 가능)
- 데이터 파이프라인 구조 증명이 목적. DB는 부가 요소

**확장 경로**: `storage.py`의 `save_documents()` 인터페이스만 유지하면 내부 구현을 pgvector로 교체 가능.
